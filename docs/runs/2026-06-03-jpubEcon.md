# Bug: JPubEcon duplicate papers across days

[[Home]] | [[Journal Scrapers]]

**Discovered:** 2026-06-03
**Status:** Fixed

---

## Symptom

Papers from JPubEcon (Journal of Public Economics) appeared in multiple consecutive daily digests — the same papers showed up on two different days.

---

## Root cause

JPubEcon was defined in **two fields** with the **same RSS URL but inconsistent watermark config:**

**`econ-political`:**
```json
{
  "name": "JPubEcon",
  "url": "https://rss.sciencedirect.com/publication/science/00472727",
  "publisher": "elsevier_general",
  "id_pattern": "pii/S(\\d+)",
  "tag_filter": null
}
```

**`econ-education`:**
```json
{
  "name": "JPubEcon",
  "url": "https://rss.sciencedirect.com/publication/science/00472727",
  "publisher": "elsevier_general",
  "tag_filter": null
}
```

`_collect_journals()` deduplicates by URL — so only one definition is used per run. **Which one wins depends on the order fields appear in `active_fields`**, which depends on which users ran that day.

- When `econ-political` wins (has `id_pattern`): **ID-based watermarking** — advances `preprint_watermarks["JPubEcon"]`
- When `econ-education` wins (no `id_pattern`): **date-based watermarking** — advances `journal_watermarks[url]`

On a run where the winning definition switches, one of the two watermarks is stale:
- Switching to date-based: date watermark was never advanced → re-fetches already-seen papers
- Switching to ID-based: `since_id = 0` (never set) → fetches entire feed again

---

## Fix

Added `"id_pattern": "pii/S(\\d+)"` to the `econ-education` definition, making both identical. ScienceDirect PII IDs are strictly monotonic so ID-based watermarking is more reliable than date-based for this publisher.

---

## Lesson

Any journal that appears in multiple fields must have **identical config** across all definitions — especially `id_pattern`. A linter that checks for URL-duplicate journals with mismatched config would catch this automatically.
